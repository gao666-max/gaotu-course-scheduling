## P1 — 我们看到了什么问题

### 不是排课慢，是复杂度已超人力边界

高途线下业务当前面临的是**多维度动态资源调度问题**，不是传统排课软件能解决的：

- **三线并行**：考研、考公、就业三条业务线独立运营，各自有不同的班型、周期和教师资源池
- **双态班型**：大班讲座（100-200人）与小班面授（15-30人）对教室、教师的要求截然不同，不能互换
- **跨校区教师共享**：名师是稀缺资源，需要在海淀、朝阳等多个校区之间调度，通勤时间成为硬约束
- **周期性需求爆发**：每年 11-12 月（考研冲刺）、3-4 月（省考冲刺），课程密度翻倍，教室需求陡增
- **频繁动态调整**：试听转化插班、教师临时请假、教室突发故障——每周都在调课，不是"排完就完了"

### 现有软件为什么搞不定

| 赛道 | 代表产品 | 核心短板 |
|------|---------|---------|
| 通用 SaaS | 校宝在线、校盈易 | 约束模型简单，面向中小机构标准化场景 |
| 在线教育 | ClassIn | 强于直播排课，线下物理教室建模不足 |
| 高校系统 | 科大讯飞、自研 | 学期制固定课表，不支持高频调课和弹性扩课 |

**高途需要的 = 高校级约束复杂度 × 教培级动态响应速度 × 自然语言交互**

市场上没有现成方案。这就是我们要做的。

## P2 — 谁在用这个系统

### 教务老师（一线排课人员）

**现状**：手动排课，单个订单平均 10 分钟，高峰期积压。排完后还要花三天打电话确认可行性。

**痛点**：
- "系统出了一个方案，但我不知道为什么这么排"——不可解释
- "我想把张老师的课调到周三下午，需要改多少地方"——不可交互
- "上次排课的教训完全没被记住，下次还是踩坑"——不积累经验

**需求**：可解释、可对话修改、可追溯历史

### 校区负责人（管理者视角）

**关注指标**：
- 教室利用率是否达标（当前行业基准约 40%，目标 65%+）
- 名师时间是否被高效利用（而非消耗在通勤上）
- 新开班从申请到排课上线需要多久
- 突发调课对整体课表的影响面有多大

**需求**：数据看板、异常预警、方案对比

### 教师

**关注**：课表是否合理（连续排课 vs 分散）、跨校区奔波是否可控、个人偏好是否被考虑

## P3 — 产品架构

```
                    教务人员 (自然语言输入)
                          │
                ┌─────────┴─────────┐
                │   AI 排课 Agent    │
                └─────────┬─────────┘
                          │
    ┌─────────────────────┼─────────────────────┐
    │                     │                     │
    ▼                     ▼                     ▼
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ 自然语言  │    │  约束决策层    │    │  知识沉淀层   │
│ 交互层    │───▶│  (OR-Tools)  │◀───│  (教务经验库)  │
└──────────┘    └──────┬───────┘    └──────────────┘
                       │
                       ▼
              ┌────────────────┐
              │  动态感知层      │
              │  (异常检测+重排)  │
              └────────┬───────┘
                       │
                       ▼
              ┌────────────────┐
              │  资源数据层      │
              │ 教师·教室·课程·时段│
              └────────────────┘
```

**四层说明**：

| 层级 | 职责 | 技术实现 |
|------|------|---------|
| 自然语言交互层 | 口语化排课指令 → 结构化约束 JSON | LLM 意图识别 + 槽位填充 |
| 约束决策层 | 硬约束满足 + 软约束多目标优化 | OR-Tools CP-SAT |
| 知识沉淀层 | 教务隐性经验 → 可执行规则权重 | 知识图谱 + RAG |
| 动态感知层 | 异常事件检测 + 增量重排 + 主动推送 | 事件驱动 + 局部求解 |

### 关键设计原则

- **LLM 负责理解，算法负责计算**——大模型不直接生成课表，避免幻觉
- **增量重排优于全局重建**——调课只动受影响的最小变量子集
- **每次人工调整都是一次学习**——系统追问原因，更新知识权重

## P4 — 核心创新（上）

### 创新一：LLM 理解 + 求解器计算，不是让大模型拍脑袋排课

**常规方案的误区**：直接让 LLM 输出课表 → 产生幻觉，教师同一时段出现在两个教室

**我们的做法**：
```
教务说: "把考研数学都排到上午，王老师只能在海淀校区"
           │
           ▼
    LLM 解析为结构化约束
    {
      "course": "考研数学",
      "constraints": {
        "time_preference": "上午",
        "teacher_lock": {"王老师": "海淀校区"}
      }
    }
           │
           ▼
    OR-Tools CP-SAT 求解
    → 保证零冲突的输出
```

**为什么这样做**：
- LLM 擅长理解模糊自然语言，提取意图和实体
- CP-SAT 擅长在严格数学约束下求最优解
- 两者各自做最擅长的事，边界清晰，不会互相干扰

**已有原型验证**：我们用 OR-Tools CP-SAT 在 2 校区 × 8 教师 × 8 教室 × 23 节课的场景下，2.88 秒求解出 OPTIMAL 课表，8 类硬约束全部满足，软约束目标值为 0。

## P5 — 核心创新（下）

### 创新二：教务经验数字化——"调整即学习"

**问题**：资深教务脑中积累了大量的排课经验：
- "张老师上午状态好，适合排数学类主课"
- "海淀到朝阳通勤至少留 2 小时缓冲"
- "考研英语冲刺班在考前两个月应该每周 3 次"

这些经验随人员离职而流失。

**我们的机制**：

```
教务手动调课 → 系统追问原因 → 解析回答 → 更新规则权重
```

举例：
```
系统: "检测到你把张老师的考研数学从下午调到了上午，为什么？"
教务: "张老师上午讲课状态好，数学这种需要集中注意力的课应该上午上"
        │
        ▼
系统解析并沉淀:
  规则: prefer_morning(teacher=张教授, course_type=数学类)
  权重: 8
  来源: 教务手动调整 #2026-07-19
```

**价值**：新教务入职时，系统已经内置了资深教务的排课经验，不再从零开始。

### 创新三：动态感知 + 主动建议，不止排课，更会"思考"

**传统系统**：排完课表就不管了，出问题等教务发现

**我们的系统**：

| 触发事件 | 系统响应 |
|---------|---------|
| 教师请假 | 自动检索可替代教师（赛道匹配 + 时段可用 + 风格相似度），生成 2-3 个替换方案并推送 |
| 教室故障 | 同校区同类型教室直接替换；若无，局部重排受影响课程 |
| 插班/新增课程 | 在现有空余时段中插入，不动已有排课 |
| 教室利用率持续偏低 | 主动建议：合并班级释放教室、开放为自习室、安排试听课 |
| 冲刺季到来 | 提前两周预警：当前教室容量是否够支撑扩容需求 |

**一句话**：系统不是等教务来问，而是主动告诉教务"你可能需要调整这里"。

## P6 — 异常场景演示：当王老师突然请假

这个案例比正常排课更能说明系统的"思考"能力。

### 场景

> 周二上午 10:00，教务收到消息：王老师（考公行测，海淀校区）因急事需请假三天。

### 传统处理方式

1. 教务打开 Excel / 排课软件查王老师的课表
2. 手动找出受影响的 3 个班、7 节课
3. 逐个打电话问其他老师能否代课
4. 手动调课、通知学生、更新财务结算
5. 耗时：半天到一天

### 我们的处理方式

```
系统自动检测: "王老师请假，影响 3 个班 × 7 节课"

自动分析替代方案:

方案 A: 孙老师替代
  - 匹配度: 92%
  - 原因: 主攻考公行测，同时段空闲，无跨校区冲突
  - 代价: 孙老师当日课时从 3 增至 4 (上限内)

方案 B: 将行测课程延期 + 调整后续安排
  - 匹配度: 60%
  - 原因: 无可完美替代教师
  - 代价: 2 个班的课程进度推迟 1 天

推荐方案 A: 学生影响最小，教师负载仍在上限内
```

教务一键确认 → 系统自动执行：
- 替换课表中的教师
- 通知受影响学生（企业微信/短信）
- 同步更新财务结算系统
- 记录本次调整原因，供未来类似场景参考

**全程耗时：不到 1 分钟。**

## P7 — MVP 落地路线

不追求一步到位，分三阶段渐进落地。

### 第一阶段：单校区智能排课助手（2-3 个月）

**目标**：在一个校区（如海淀）验证核心技术链路

**交付物**：
- CP-SAT 约束求解核心，覆盖 8 类硬约束 + 4 类软约束
- 自然语言指令输入 → 结构化约束 → 求解 → 课表输出
- 基础课表可视化（周视图）

**验证指标**：
- 排课耗时从 3 天 → 2 小时
- 硬约束冲突 = 0
- 教务可接受方案率 ≥ 85%（首次输出即可用，无需大幅手动调整）

### 第二阶段：知识沉淀 + 异常检测（2-3 个月）

**目标**：让系统"学会"教务的经验，能主动发现问题

**交付物**：
- "调整即学习"机制（教务手动调课 → 追问原因 → 更新规则权重）
- 教师请假/教室故障异常检测 + 替代方案推荐
- 教务经验知识库

**验证指标**：
- 教务手动调整次数逐月下降 30%
- 异常检测覆盖率 ≥ 90%

### 第三阶段：多校区全局调度（2-3 个月）

**目标**：跨校区联合优化，支持弹性扩缩

**交付物**：
- 多校区资源池统一建模
- 跨校区教师调度优化（含通勤时间约束）
- 冲刺季自动扩容 + 淡季资源回收建议

**验证指标**：
- 跨校区教师通勤总时间下降 20%
- 教室综合利用率从 40% → 65%+

## P8 — 约束建模核心（简化版）

### 排课变量定义

| 变量 | 含义 | 取值范围 |
|------|------|---------|
| C | 课程集合 | 考研/考公/就业 × 大班/小班 |
| T | 教师集合 | 含赛道属性、校区属性、时间偏好 |
| R | 教室集合 | 含容量、类型(大讲堂/小教室/机房/面试间)、校区 |
| S | 时段集合 | 5天 × 4时段 = 20 时段/周 |

### 硬约束（必须满足，违反即不可行）

| # | 约束 | 建模表达 |
|---|------|---------|
| H1 | 教室容量 ≥ 学生数 | `∀c∈C: room_capacity ≥ c.students` |
| H2 | 教室类型匹配 | `∀c∈C: room_type ∈ c.allowed_types` |
| H3 | 教室同时段唯一 | `∀r∈R, ∀s∈S: sum(X[*,r,s]) ≤ 1` |
| H4 | 教师同时段唯一 | `∀t∈T, ∀s∈S: sum(X[*,*,s] where teacher=t) ≤ 1` |
| H5 | 教师资质匹配 | 教师赛道 ∩ 课程赛道 ≠ ∅ |
| H6 | 跨校区缓冲 | 相邻时段跨校区 → 禁止 |
| H7 | 日课时上限 | `∀t∈T, ∀d: sum(X[t,*,d,*]) ≤ max_daily` |
| H8 | 课程分散 | 同一课程不同 session 不在同一天 |

### 软约束（加权优化）

| # | 约束 | 权重 | 说明 |
|---|------|------|------|
| S1 | 教师赛道偏好 | 10 | 优先匹配主攻赛道 |
| S2 | 连续排课偏好 | 8 | 避免课表空洞 |
| S3 | 跨校区惩罚 | 15 | 同天跨校区给高惩罚 |
| S4 | 班型时段偏好 | 5 | 大班上午/小班灵活 |

### 目标函数

```
Minimize: 10×Σ(跨赛道教学) + 8×Σ(课表空洞) + 15×Σ(同天跨校区) + 5×Σ(时段错配)
Subject to: H1-H8 全部硬约束
```

## P9 — 预期价值与可行性

### 可量化指标

| 指标 | 当前行业基准 | 目标值 | 依据 |
|------|------------|--------|------|
| 排课耗时 | 3-5 天 | < 2 小时 | 新东方合肥部署后实测 |
| 教室利用率 | ~40% | 65%+ | 命题要求 + CP-SAT 求解验证 |
| 硬约束冲突率 | 人工易出错 | 0 | 约束求解器数学保证 |
| 调课响应时间 | 半天-1天 | < 1 分钟 | Agent 自动检测 + 方案推送 |
| 教务手动调整次数 | 每次排课大量调整 | 逐月下降 30% | 知识沉淀机制持续学习 |

### 技术可行性

- **OR-Tools CP-SAT**：Google 开源维护，工业级稳定性。巴西利亚大学 11 个真实场景全部成功求解（IEEE ITHET 2024 最佳论文）。我们在原型中验证：23 节课 2.88 秒 OPTIMAL 求解。
- **LLM 意图识别**：当前大模型在结构化信息抽取任务上成熟可靠，不作为生成引擎而作为解析引擎，不产生幻觉风险。
- **渐进落地**：单校区 MVP 可在 2-3 个月内跑通并验证，不依赖大规模基础设施投入。

### 可推广性

- **高途内部**：三条业务线（考研/考公/就业）共享核心约束框架，一个校区验证后横向复制
- **行业推广**：我国 65% 的教培企业仍依赖"系统+Excel"混合作业（华为云 2026 年报告），本方案可横向覆盖职业培训、素质教育、企业培训等多赛道
